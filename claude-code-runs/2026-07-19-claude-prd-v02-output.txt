Le PRD v0.2 est en place dans `prd/PRD.md`. J'ai réécrit le document en conservant tout ce qui restait valable de la v0.1 (personas, parcours en 10 étapes, exigences RGPD/AI Act/cybersécurité, discipline d'exécution) et en le faisant pivoter du « dépôt documentaire » vers « documentation + application web fonctionnelle dès la V1 ». Aucun autre fichier n'a été créé ni modifié.

Les changements structurants :

- **Cartouche et note de révision.** Version 0.2 datée du 19 juillet 2026, avec un encadré « Ce qui change entre v0.1 et v0.2 » qui assume explicitement la correction : l'interface web n'est plus repoussée en V3, elle est au cœur de la V1.
- **Promesse (§1.3).** La formulation proposée dans le brief est reprise mot pour mot, suivie des deux dimensions du dépôt : couche pédagogique générique (définition, cycle de vie, rôles, classification, RGPD/AI Act/cyber, tests, réversibilité, trois statuts) et premier harnais vertical complet, l'onboarding RH documentaire.
- **« Ce que le produit n'est pas » (§1.4).** Section durcie : pas un SIRH, pas un « quasi-SIRH » (l'expression n'apparaît que là, pour être écartée), pas de dossiers agents, pas de workflow décisionnel, pas de paie/carrière/absences/temps/santé/discipline/rémunération, pas de décision individuelle. Les formulations recommandées (« application d'onboarding RH documentaire », « portail d'accueil documentaire pour nouveaux arrivants ») y sont fixées.
- **MVP (§6).** Les douze fonctions attendues sont contractualisées une à une (accueil, parcours, fiches, FAQ sourcée, quiz, checklist RH, sources, limites et refus, gouvernance, mode démo, mode configuration, tests de garde-fous), avec une section sur le comportement du moteur documentaire (sourçage exclusif, refus des cas individuels, mentions systématiques, interface modèle substituable, mode dégradé sans clé API — utile pour la vidéo) et le parcours de réplication en six commandes tel que demandé.
- **Architecture (§7).** Next.js ou équivalent recommandé sans être verrouillé (point à confirmer en V0), fichiers Markdown/YAML/JSON, pas de base de données en V1, exécution locale, design sobre et institutionnel, déploiement optionnel et postérieur. J'ai ajouté une sous-section « ce que l'architecture refuse » pour barrer la route à toute stack lourde en phase d'implémentation.
- **Structure du dépôt (§5).** Nouvelle arborescence conforme au brief (`src/`, `content/demo-onboarding-rh/`, `configs/`, `docs/`, `templates/`, `tests/guardrails/` + `tests/structure/`, `scripts/`), avec un invariant énoncé : documentation + application + contenu démo + templates + tests, et la frontière contenu/code posée comme engagement produit — adapter le portail ne demande jamais de toucher `src/`.
- **Roadmap (§11).** Recalée : V0 PRD v0.2 validé → V1 webapp fonctionnelle + documentation → V1.1 configuration guidée → V1.2 tests renforcés + scénario vidéo → V2 observatoire territorial → V3 déploiement simplifié / guides OPSN → V4 agents avancés si retour terrain.
- **Plan de développement Opus 4.8 (§12).** Redécoupé en huit lots A–H (socle, app minimale, contenu RH, UI pédagogique, moteur documentaire, garde-fous, tests, documentation/vidéo), en gardant la règle « une tâche = portée fermée + vérification » et l'interdiction de dériver vers des fonctions de gestion RH.
- **Plan vidéo (§13).** Dix séquences alignées sur le brief : clone, lancement local, visite du portail, réponse sourcée, question piège refusée (séquence pivot), pages sources/limites/gouvernance, adaptation par une collectivité, `npm test` en direct, conclusion « ce n'est pas un SIRH, c'est un harnais documentaire gouverné ».
- **Exigences de sérieux (§9).** Renforcées et rendues doubles : chaque garantie documentée doit être visible dans l'application (classification affichée, sources datées à l'écran, journal de mise à jour, mention « ne vaut pas validation juridique », responsable métier et DPO/DSI/RSSI dans le workflow de validation) et vérifiée par un test.

Le document ne mentionne aucun concurrent ni fork, et rappelle que le développement n'interviendra qu'après validation humaine, avec Opus 4.8 (identifiant à reconfirmer au lancement, sans substitution silencieuse). Un point à confirmer a été ajouté pour l'arbitrage de la stack web définitive en V0.
